# Data Modeling (We will cover `Relations` in this section)

MongoDB is a NoSQL database, it doesn't enforce strict schema relationships like foreign keys in relational databases.

We can still model relationships between documents in MongoDB using a few approaches.  
The two main types of relationships are:

1. Embedded Documents (Denormalization)
2. Referenced Documents (Normalization)

The types of realtionships

- One to One
- One to Many
- Many to Many

### 2. Using `Referenced Documents (Normalization)` to handle same `One to Many Relationship`

Syntax

`users`

```js
    "email": "amit.sharma@example.com",
    "phone": "+91-987654210"
    "address": "MG Road, Mumbai, Maharashtra"

{
    "_id": "user2",
    "name": "Priya Verma",
    "email": "priya.verma@example.com",
    "phone": "+91-987654211",
    "address". "Nehru Place, New Delhi, Delhi"
}
```

`orders`

```js
{
    "_id": "orderl",
    "user_id": "user1",
    "product": "Laptop",
    "amount": 50000,
    "order_date": "2024-08-01"
},
{
    "_id": "order?"
    "user_id": "user?",
    "product": "Mobile Phone",
    "amount": 15000,
    "order_date": "2024-08-05"
}
```

Here these both documents are connected using `user_id` which is common in both documents

# JOIN with `$lookup`

- We use `lookup` to join two different documents and establish relation

```js
db.users.aggregate([
  {
    $lookup: {
      from: "orders", // The target collection to join with
      localField: "_id", // The field from the 'users' collection
      foreignField: "user_id", // The field from the 'orders' collection
      as: "orders", // The name of the new array field to add to the 'users'
    },
  },
]);
```

# Important Notes

## 1. Embedded Documents (Denormalization)

1. Document Size Limit

   MongoDB imposes a maximum documents size limit of `16MB`. This means that the entire document, including all embedded documents, `cannot exceed 16MB in size`.  
   This limit is generally large enough for most use cases, but it can become a concern if you have a lot of embedded data.

2. Nesting Depth Limit

   MongoDB allows documents to be nested up to `100 levels deep`. However, deeply nested documents can become difficult to manage and may lead to performance issues, so it is generally advisable to keep the nesting as shallow as possible.

## 2. Referenced Documents (Normalization)

1. Document Size Advantage

   With referenced documents, `each document only stores references` (usually ObjectIDs or foreign keys) to related documents instead of embedding all the data directly.  
   This means the `16MB document size limit is less of a concern, since related data lives in separate documents`. Large or growing datasets can be spread across multiple collections, avoiding oversized documents.

2. Nesting Depth Advantage

   MongoDB’s `100-level nesting limit does not apply the same way with references`, since referenced documents are linked but not physically nested inside each other.  
   Instead, relationships are managed through separate documents in different collections. However, `this comes with the tradeoff` that queries often `require additional lookups or $lookup aggregations` to bring the data together, `which may affect performance`.
