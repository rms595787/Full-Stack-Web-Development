# Schema Validation in MongoDB

Schema validation is a mechanism in MongoDB that allows you to enforce specific rules on the structure and contents of documents in a collection. These rules can include required fields, data types, value ranges, or patterns.

- MongoDB uses a JSON Schema format to define the validation rules.
- Allows you to specify various constraints and rules for your documents, such as required fields, field types, and value ranges.

## Need for Schema Validation

1. `Data Consistency`
   - MongoDB is schema-less by default, which means documents in a collection can have varying structures.
   - Without validation, inconsistent or malformed data may get inserted.
   - Schema validation ensures all documents follow a uniform structure.
2. `Data Integrity`
   - Helps maintain the correctness of data by enforcing constraints (e.g., age must be a number, email must match a regex pattern).
   - Prevents invalid or incomplete data from being stored.
3. `Error Prevention`
   - Reduces bugs caused by unexpected document structures in application code.
   - Ensures the database itself rejects incorrect data before it propagates to the application layer.
4. `Improved Query Reliability`
   - Queries rely on predictable structures.
   - With schema validation, developers can query collections with confidence that fields exist and are of the expected type.
5. `Collaboration & Scalability`
   - In large teams, schema validation acts as a contract, ensuring everyone inserts data in a consistent format.
   - Makes it easier to scale applications without unexpected data anomalies.

## Implementation

### 1. How to use Schema Validation?

- We can add validation while creating collection...

```js
db.createCollection("user1", {
  validator: {
    $jsonSchema: {
      bsonType: "object",
      required: ["name", "phone"],
    },
  },
});
```

<img src="./_images/Schema Validation1.png" width=600>

Here we can see when we try to insert only name, it throughs error that `missingProperties: [ "phone" ]`

```js
db.createCollection("users", {
  validator: {
    $jsonSchema: {
      bsonType: "object",
      required: ["name", "age"],
      properties: {
        name: {
          bsonType: "string",
          description: "must be a string and is required",
        },
        age: {
          bsonType: "int",
          minimum: 18,
          description: "must be an integer and is required",
        },
      },
    },
  },
  validationLevel: "strict",
  validationAction: "error",
});
```

### 2. Update Schema for Existing Collection

If a collection already exists, use `db.runCommand` and `collMod` (collection modification) to apply schema validation:

```js
db.runCommand({
  collMod: "users",
  validator: {
    $jsonSchema: {
      bsonType: "object",
      required: ["name", "age", "phone"],
      properties: {
        name: {
          bsonType: "string",
          description: "must be a string and is required",
        },
        age: {
          bsonType: "int",
          minimum: 18,
          description: "must be an integer and is required",
        },
        phone: {
          bsonType: "int",
          minimum: 0,
          maximum: 10,
          description: "must be an integer and is required",
        },
      },
    },
  },
  validationLevel: "strict",
  validationAction: "error",
});
```

## Validation Options

### validationLevel

1. `strict` → applies validation to all inserts/updates (default).
2. `moderate` → only validates new/updated fields (not existing invalid docs).

3. Strict → The document must fully comply with the schema validation rules. If a document does not comply, it will not be inserted or updated in the collection.

4. Moderate → Only new documents and modifed fields in existing documents are validated against the schema. This allows for partial validation and can be useful for legacy systems or gradual schema enforcement.

### validationAction

1. `error` → rejects invalid documents (default).
2. `warn` → allows invalid docs but logs a warning.

3. error → If a document does not meet the schema validation criteria, MongoDB will throw an error and reject the insert or update operation.

4. warn → MongoDB logs a warning message when a document does not meet the schema validation criteria but still allows the insert or update operation.

# To identify which schema collection is currently following

🔹 How to Identify Collection Schema Validation
MongoDB stores validation rules in the collection options.
You can inspect them using these commands:

1. Using listCollections

   ```js
   db.getCollectionInfos({ name: "users" });
   ```

   👉 This will return something like:

   ```js
   [
     {
       name: "users",
       type: "collection",
       options: {
         validator: {
           $jsonSchema: {
             bsonType: "object",
             required: ["name", "age", "email"],
             properties: {
               name: { bsonType: "string" },
               age: { bsonType: "int" },
               email: { bsonType: "string" },
             },
           },
         },
         validationLevel: "strict",
         validationAction: "error",
       },
     },
   ];
   ```

   <img src="./_images/Schema Validation3.png" width=600>

2. Using db.getCollectionInfos() (all collections)

   ```js
   db.getCollectionInfos();
   ```

   This lists all collections with their validation rules.

   <img src='./_images/Schema Validation4.png' width=600>
   <img src='./_images/Schema Validation5.png' width=600>

3. Using db.runCommand({ listCollections: 1 })

   ```js
   db.runCommand({ listCollections: 1, filter: { name: "users" } });
   ```

   This also shows validator details if present.

   <img src='./_images/Schema Validation6.png' width=600>

✅ If you see a $jsonSchema block → the collection is following schema validation rules.
❌ If validator is missing or {} → the collection has no schema validation, and is fully flexible (default MongoDB behavior).
