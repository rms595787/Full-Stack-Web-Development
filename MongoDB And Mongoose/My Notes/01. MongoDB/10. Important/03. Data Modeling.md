# MongoDB – Data Modeling

## 📂 Embedding vs Referencing

### 🔹 Embedding (denormalization)

- Store related data inside the same document.
- Good for **one-to-few** relationships.
- Faster reads, fewer joins.

```json
{
  "name": "Alice",
  "orders": [
    { "product": "Book", "qty": 1 },
    { "product": "Pen", "qty": 2 }
  ]
}
```

### 🔹 Referencing (normalization)

- Store references (ObjectIds).
- Good for one-to-many or many-to-many relationships.
- More flexible, avoids document size limit (16 MB).

```json
// User document
{ "_id": ObjectId("u1"), "name": "Alice" }

// Order document
{ "_id": ObjectId("o1"), "userId": ObjectId("u1"), "product": "Book" }
```

### 📑 Common Patterns

| Pattern                 | Use Case                          | Example                                                                 |
| ----------------------- | --------------------------------- | ----------------------------------------------------------------------- |
| One-to-Few (Embed)      | User + few addresses              | `{ name: "Bob", addresses: [ {...}, {...} ] }`                          |
| One-to-Many (Reference) | User + many orders                | `{ userId: ObjectId("u1"), ... }`                                       |
| Many-to-Many            | Students & Courses                | Linking collection                                                      |
| Bucket Pattern          | Time-series data                  | Store multiple readings per doc                                         |
| Polymorphic             | Different types in one collection | `{ type: "car", brand: "BMW" }` and `{ type: "bike", brand: "Yamaha" }` |

# ⚡ Best Practices

- Embed when data is mostly read together.
- Reference when data grows unbounded.
- Keep documents under 16 MB.
- Index referencing fields (userId, postId).
