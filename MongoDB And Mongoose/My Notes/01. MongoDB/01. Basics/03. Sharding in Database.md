# 📝 Sharding in Databases

## 🔹 Definition

Sharding is a database scaling technique where data is horizontally partitioned (split row-wise) across multiple smaller, faster databases (called shards).

- Each shard holds a subset of the data.
- All shards together make up the complete dataset.
- Shards can live on different servers, improving performance, scalability, and availability.

## 🔹 Example

Imagine a **user database** with millions of records:

- Instead of keeping all users in a single DB table (which becomes slow),
- You **shard** users by their `user_id`:

| Shard   | User IDs Stored       | Server Location |
| ------- | --------------------- | --------------- |
| Shard 1 | 1 – 1,000,000         | Server A        |
| Shard 2 | 1,000,001 – 2,000,000 | Server B        |
| Shard 3 | 2,000,001 – 3,000,000 | Server C        |

Now queries for user_id = 1,500,000 go only to Shard 2, not the entire DB.

## 🔹 Advantages

- ✅ Handles **very large datasets**
- ✅ Improves **query performance** (parallel reads/writes)
- ✅ Increases **availability** (failure in one shard doesn’t bring down all)
- ✅ Enables **horizontal scaling** (add more shards/servers as data grows)

## 🔹 Challenges

- ⚠️ Complex to implement and maintain
- ⚠️ Rebalancing shards when data grows unevenly
- ⚠️ Cross-shard queries (e.g., joins across shards) are expensive
- ⚠️ Application logic often needs to be **shard-aware**

## 🔹 Real-world Examples

- **Instagram / Facebook** → user data sharded by user_id
- **YouTube** → videos and metadata sharded across servers
- **E-commerce sites** → orders sharded by region or customer_id

✅ In short:

- Sharding = `splitting a big database into smaller, distributed databases (shards) for performance and scalability.`
