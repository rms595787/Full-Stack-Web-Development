## 1. What is a Role in PostgreSQL?

- A **role** is a database entity that can **own database objects** (tables, views, functions) and **have privileges**.
- A role can function as:
  - **User role (with LOGIN)** → Can log in with a username + password.
  - **Group role (without LOGIN)** → Can’t log in, but can hold permissions and be assigned to users.

💡 In PostgreSQL, **users and groups are unified as roles**.

---

## 2. Role Attributes (Capabilities)

When creating a role, we can assign **attributes**.

| Attribute       | Description                                                       |
| --------------- | ----------------------------------------------------------------- |
| **LOGIN**       | Allows role to log in (be a user).                                |
| **SUPERUSER**   | Full control over DB, bypasses permission checks.                 |
| **CREATEDB**    | Can create new databases.                                         |
| **CREATEROLE**  | Can create/modify other roles.                                    |
| **INHERIT**     | Role inherits privileges from roles it’s a member of. _(default)_ |
| **NOINHERIT**   | Must explicitly use `SET ROLE` to use group privileges.           |
| **REPLICATION** | Can initiate streaming replication.                               |
| **BYPASSRLS**   | Bypass row-level security policies.                               |

---

## 3. Creating Roles

### ✅ Create a Login Role (user):

```sql
CREATE ROLE dev_user WITH LOGIN PASSWORD 'mypassword';
```

### ✅ Create a Group Role:

```sql
CREATE ROLE readonly_group;
```

### ✅ Create a Superuser:

```sql
CREATE ROLE admin WITH SUPERUSER LOGIN PASSWORD 'adminpass';
```

---

## 4. Managing Roles

### 🔹 Alter Role

```sql
ALTER ROLE dev_user CREATEDB;       -- grant ability to create DB
ALTER ROLE dev_user NOCREATEDB;     -- revoke
ALTER ROLE dev_user PASSWORD 'newpass'; -- change password
ALTER ROLE dev_user VALID UNTIL '2025-12-31'; -- expiry
```

### 🔹 Drop Role

```sql
DROP ROLE dev_user;
```

### 🔹 Rename Role

```sql
ALTER ROLE dev_user RENAME TO developer;
```

---

## 5. Role Membership (Groups)

Roles can be members of other roles → similar to UNIX groups.

```sql
GRANT readonly_group TO dev_user;
```

- Now `dev_user` has all privileges of `readonly_group`.
- By default, privileges are **inherited** (unless `NOINHERIT` is set).

Check memberships:

```sql
\du
```

---

## 6. Role Privileges (Permissions)

Roles can be given permissions at different levels:

### A. Database-Level Privileges

```sql
GRANT CONNECT ON DATABASE mydb TO dev_user;
GRANT CREATE ON DATABASE mydb TO dev_user;
```

### B. Schema-Level Privileges

```sql
GRANT USAGE ON SCHEMA public TO dev_user;
GRANT CREATE ON SCHEMA public TO dev_user;
```

### C. Table-Level Privileges

```sql
GRANT SELECT, INSERT, UPDATE, DELETE ON mytable TO dev_user;
```

### D. Sequence Privileges

```sql
GRANT USAGE, SELECT, UPDATE ON mysequence TO dev_user;
```

### E. Function/Procedure Privileges

```sql
GRANT EXECUTE ON FUNCTION myfunc() TO dev_user;
```

### F. ALL PRIVILEGES (shortcut)

```sql
GRANT ALL PRIVILEGES ON mytable TO dev_user;
```

---

## 7. Revoking Privileges

```sql
REVOKE INSERT ON mytable FROM dev_user;
REVOKE ALL PRIVILEGES ON DATABASE mydb FROM dev_user;
```

---

## 8. Default Privileges

Set default privileges for new objects:

```sql
ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT ON TABLES TO readonly_group;
```

👉 Ensures new tables automatically get read permissions for `readonly_group`.

---

## 9. Security Best Practices

1. **Avoid using `SUPERUSER`** except for admins.
2. Create **separate roles for applications** (principle of least privilege).
3. Use **group roles** for shared permissions (e.g., `readonly`, `readwrite`).
4. Use **row-level security (RLS)** for fine-grained access control.

## 10. Inspecting Roles

- List all roles:
  ```sql
  \du
  ```

- Check role memberships:
  ```sql
  \dg+
  ```
- Show current user:
  ```sql
  SELECT current_user;
  ```
- Show session user:
  ```sql
  SELECT session_user;
  ```

## 11. Special Predefined Roles

PostgreSQL provides built-in roles (useful in permissions):

- **pg_read_all_data** → Can `SELECT` all tables.
- **pg_write_all_data** → Can `INSERT/UPDATE/DELETE` on all tables.
- **pg_monitor** → For monitoring.
- **pg_signal_backend** → Can send signals to other backends.
- **pg_execute_server_program** → Can run programs with `COPY FROM PROGRAM`.

Example:

```sql
GRANT pg_read_all_data TO dev_user;
```

## 12. Example Setup

```sql
-- Create roles
CREATE ROLE readonly_group;
CREATE ROLE writer_group;
CREATE ROLE app_user LOGIN PASSWORD 'app_pass';

-- Grant privileges
GRANT CONNECT ON DATABASE mydb TO app_user;
GRANT USAGE ON SCHEMA public TO app_user;

-- Group roles
GRANT readonly_group TO app_user;
GRANT writer_group TO app_user;

-- Privileges
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_group;
GRANT INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO writer_group;
```

---

✅ **Summary:**

- **Roles = users + groups.**
- Roles can have **attributes** (`LOGIN`, `SUPERUSER`, etc.).
- You can **GRANT / REVOKE privileges** at database, schema, table, function, and sequence levels.
- You can organize access with **group roles** and **default privileges**.
- Always follow **least privilege principle**.
