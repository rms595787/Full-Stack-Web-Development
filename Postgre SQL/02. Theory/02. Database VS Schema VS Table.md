# 1. Database

- **What it is:**

  The **highest-level container** in PostgreSQL (or any RDBMS).  
  Think of it as a **separate project/environment** where all schemas, tables, and data live.

- **Key point:**

  One PostgreSQL instance can have **multiple databases** (e.g., hr_db, sales_db, finance_db).

- **Real-life IT example:**

  - In a company like **Amazon**, they may have separate databases for:

    - **Orders Database** → stores all purchase orders.
    - **Customer Database** → stores customer details and preferences.
    - **Inventory Database** → stores product and stock information.

  Each database is independent — no direct table sharing across them unless you use **foreign data wrappers** or APIs.

# 2. Schema

- **What it is:**
  A logical namespace inside a database.  
   It groups related objects like tables, views, functions, and sequences.
  Schemas help avoid name conflicts and organize data better.
- **Key point:**
  One database can have multiple schemas (e.g., public, hr, analytics).
- **Real-life IT example:**

  - In a Banking Application (finance_db):

    - customers schema → tables related to customer profiles, accounts.
    - transactions schema → tables for payments, transfers, loans.
    - compliance schema → tables for audit logs, KYC details.

This separation ensures that different teams (compliance team, customer service team, transaction monitoring team) can work in the same database but in their own schemas without stepping on each other.

# 3. Table

- **What it is:**

  The lowest-level object where actual data is stored in rows and columns.  
  Each table belongs to one schema.

- **Key point:**

  Tables hold the data records — just like Excel sheets.

- **Real-life IT example:**

  - In an E-commerce Database (orders_db → orders schema):

    - orders table → stores order_id, customer_id, product_id, order_date.
    - order_items table → stores details of each item in the order.
    - shipping table → stores delivery address, courier details, tracking info.

These tables are queried together using joins to generate reports like “all orders placed in the last 24 hours that are pending shipment.”

# Example 1: E-Commerce System

- **Database:** ecommerce_db
- **Schemas:**

  - customers → stores customer-related data.
  - orders → stores order and shipping data.

- **Tables:**

  - customers.customer_info → customer_id, name, email.
  - orders.order_details → order_id, customer_id, product_id.
  - orders.shipping → order_id, address, status.

```yaml
Database: ecommerce_db
|
|-- Schema: customers
|     |-- Table: customer_info (customer_id, name, email)
|
|-- Schema: orders
      |-- Table: order_details (order_id, customer_id, product_id)
      |-- Table: shipping (order_id, address, status)
```

# Example 2: Ride-Sharing App (Uber/Ola)

- **Database:** rides_db
- **Schemas:**

  - drivers → data about drivers.
  - customers → data about riders.
  - trips → data about trips and payments.

- **Tables:**

  - drivers.driver_profiles → driver_id, name, license_no.
  - customers.customer_profiles → customer_id, name, phone.
  - trips.trip_details → trip_id, driver_id, customer_id, start_time, end_time.
  - trips.trip_payments → payment_id, trip_id, amount.

```yaml
Database: rides_db
|
|-- Schema: drivers
|     |-- Table: driver_profiles (driver_id, name, license_no)
|
|-- Schema: customers
|     |-- Table: customer_profiles (customer_id, name, phone)
|
|-- Schema: trips
      |-- Table: trip_details (trip_id, driver_id, customer_id, start_time, end_time)
      |-- Table: trip_payments (payment_id, trip_id, amount)
```
