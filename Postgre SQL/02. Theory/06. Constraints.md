# SQL Constraints and Keywords Cheat Sheet

Constraints are **rules applied to table columns** to enforce data integrity.  

---

| Constraint / Keyword | Description | Example |
|----------------------|------------|---------|
| **PRIMARY KEY** | Uniquely identifies each record in a table; automatically `NOT NULL` | `id INT PRIMARY KEY` |
| **FOREIGN KEY** | Ensures referential integrity by linking to a primary key in another table | `dept_id INT REFERENCES department(id)` |
| **UNIQUE** | Ensures all values in a column are unique | `email VARCHAR(100) UNIQUE` |
| **NOT NULL** | Ensures a column cannot have `NULL` values | `name VARCHAR(50) NOT NULL` |
| **CHECK** | Ensures values satisfy a boolean expression | `salary NUMERIC CHECK(salary > 0)` |
| **DEFAULT** | Sets a default value if none is provided | `status VARCHAR(10) DEFAULT 'active'` |
| **AUTO_INCREMENT / SERIAL / BIGSERIAL** | Automatically increments numeric column (PostgreSQL uses `SERIAL`/`BIGSERIAL`) | `id SERIAL PRIMARY KEY` |
| **ON DELETE CASCADE** | Deletes dependent rows when referenced row is deleted | `FOREIGN KEY(dept_id) REFERENCES department(id) ON DELETE CASCADE` |
| **ON UPDATE CASCADE** | Updates dependent rows when referenced key changes | `FOREIGN KEY(dept_id) REFERENCES department(id) ON UPDATE CASCADE` |
| **DEFERRABLE / NOT DEFERRABLE** | Determines if constraint checking can be deferred to transaction commit | `CONSTRAINT fk_dept FOREIGN KEY(dept_id) REFERENCES department(id) DEFERRABLE INITIALLY DEFERRED` |
| **INDEX** | Creates an index to speed up lookups (not exactly a constraint but related to performance) | `CREATE INDEX idx_name ON person(name);` |
| **PRIMARY KEY (column1, column2)** | Composite primary key (multiple columns) | `PRIMARY KEY(id, dept_id)` |
| **UNIQUE (column1, column2)** | Composite unique key | `UNIQUE(email, phone)` |
| **EXCLUDE** | PostgreSQL-specific, ensures uniqueness using operators | `EXCLUDE USING gist (ts WITH &&)` |
| **CHECK (expression)** | Column must satisfy expression | `CHECK(age >= 18)` |

---

### ✅ Notes:

1. **PRIMARY KEY vs UNIQUE**: Both enforce uniqueness, but a table can have **only one primary key** but multiple unique constraints.  
2. **FOREIGN KEY**: Links tables and ensures referential integrity; can have `ON DELETE`/`ON UPDATE` rules.  
3. **SERIAL / BIGSERIAL**: PostgreSQL-specific auto-increment types.  
4. **CHECK constraints**: Useful for business rules like salary > 0, age ≥ 18, etc.  
5. Composite keys (PRIMARY or UNIQUE) are useful when uniqueness is defined across multiple columns.
